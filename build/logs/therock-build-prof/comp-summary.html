General FAQ

  1. rss = Resident Set Size

    It means the amount of physical RAM actually in use by a process at its peak. High rss means memory-heavy compilation or linking. If rss is high across many parallel jobs, we can hit swapping, cache thrashing, OOM kills in CI. High rss often limits how much parallelism you can safely use (-j)

  2. max_rss_mb

    the maximum RAM a compiler or linker process had allocated in memory at any point.

  3. Wall time sum vs component span vs estimated elapsed

    wall_time_sum_min is the sum of wall times across all commands for that component. In parallel builds this can be far larger than the total build duration because many commands run concurrently.

    wall_time_span_min is the component’s elapsed span: max(end_time) - min(start_time) across all commands belonging to that component. This is usually much closer to “how long the build spent working on that component”, though it can still overlap with other components (parallelism).

    wall_time_est_elapsed_min is an estimated elapsed contribution computed from average build concurrency: we compute avg_concurrency = sum(real_s) / build_span_s using timestamps, and then estimate: wall_time_est_elapsed_min ≈ wall_time_sum_min / avg_concurrency.

    That doesn’t mean the build is slower; it means processes are spending time waiting (I/O, scheduling, throttling, contention), so CPU time isn’t keeping up with wall time.

  4. What Avg Threads (avg_threads) actually mean ?

    Case 1> avg_threads ≈ 1.0

    This means process used about one CPU core for most of its runtime.

    Case 2> avg_threads < 1.0

    Meaning: The process spent a lot of time waiting, not computing (I/O, contention, throttling).

    Case 3> avg_threads > 1.0

    Meaning: The process used multiple CPU cores simultaneously (e.g. LTO backends, LLVM worker threads).

  5. What different sums mean ?

    user_sum_min → time spent executing your code (compiler, linker, optimizer logic)

    If this is high:

    the build is compute-heavy

    faster CPUs, fewer templates, or fewer TUs help

    more parallelism may help if avg_threads > 1

    sys_sum_min → time spent inside the operating system kernel

    If this is high:

    you’re often I/O-bound

    disk speed, filesystem, caching, or build directory layout matters

    adding more CPUs will not help much

    cpu_sum_min → total CPU time = user_sum_min + sys_sum_min

    How much CPU did this component cost overall?

    It’s the best metric for:

    capacity planning

    CI cost estimation

    “what’s expensive” comparisons between components

Key mental model

  • wall_time_sum_min → total summed command wall time (inflates with parallelism)
  • wall_time_span_min → component elapsed window (closest to “actual component time”)
  • wall_time_est_elapsed_min → concurrency-adjusted estimate
  • cpu_sum_min → cost/compute spent
  • avg_threads → CPU utilization ratio per component
  • RSS → limits safe parallelism
