# TheRock Debug Symbol Generation & Packaging

## Overview

TheRock currently ships release binaries with little to no usable host-side debug
information. Field debugging, crash triage, and profiling of ROCm components therefore
requires rebuilding from source with a custom configuration, which is slow and rarely
reproduces the exact shipped binary. This RFC defines a single, cross-platform
mechanism capable of **generating and packaging host-side debug symbols** for every
CMake subproject. Global defaults and per-subproject controls let release configurations
select symbol coverage without changing build types.

The design deliberately builds on infrastructure that already exists in the tree
(`THEROCK_SPLIT_DEBUG_INFO`, the Linux `.build-id` split pipeline, the `dbg` artifact
component, and the MSVC embedded-debug configuration) rather than introducing a parallel
system. Its central requirement is that **Release builds remain Release builds**: debug
information is added without downgrading optimization to `RelWithDebInfo`.

Our goals are to:

1. **Generate host-side debug symbols for all supported operating systems** in their
   native formats (DWARF/`.debug` on Linux, PDB on Windows).
1. **Support every CMake subproject**, including `rocm-systems`, `rocm-libraries`,
   CMake-built system dependencies, and the bootstrapped compiler, while allowing each
   subproject's symbol generation to be controlled independently.
1. **Provide global defaults with per-component overrides** so release configurations
   can control symbol volume without changing build types or package contents. In
   particular, release presets can opt large components such as `amd-llvm` out with
   `-Damd-llvm_GENERATE_DEBUG_INFO=OFF`; this is preset policy, not a hardcoded default,
   and users can override it.
1. **Package debug symbols as separate, opt-out-able deliverables** that never bloat
   the runtime or development packages.
1. **Enforce one common, OS-agnostic CMake mechanism** so that every configuration
   (including `RelWithDebInfo`) routes through the same code path.
1. **Preserve build performance**, including compatibility with `ccache`/`sccache`.

## Scope

### In Scope

- Host-side debug symbol generation on all OSes TheRock supports.
- Native debug formats: Linux ELF/DWARF separate `.debug` files (build-id indexed),
  Windows PDB.
- Debug-symbol generation support for all CMake subprojects across `base`, `core`, `rocm-systems`,
  `rocm-libraries`, `third-party/sysdeps`, and the bootstrapped compilers
  (`amd-llvm`, `amd-comgr`, `hipcc`).
- Debug-symbol level control (`minimal`/`full`/`extra` on Linux and
  `minimal`/`full` on Windows; public vs. private PDB).
- Per-component overrides for generation, detail level, and splitting, following the
  same naming model as `{project}_BUILD_TYPE`.
- Separate debug packaging for tarball and native (deb/rpm) channels.
- `ccache`/`sccache` compatibility for debug-info-bearing compilation.
- Release-preset updates and wiring those settings into active nightly/release CI
  workflows.

### Out of Scope

- **GPU kernel / device-code debug symbols.** Only host code is covered.
- **Python debug-symbol packaging and distribution**, including symbols for native
  libraries generated by Python wheels and native libraries shipped inside `rocm_sdk`.
  Matching symbols to exact wheel binaries and selecting a delivery mechanism require a
  separate follow-up design, potentially together with symbol-server support.
- **Source-level packaging** (source packages are covered by RFC0009).
- **Symbol-server / debuginfod hosting and delivery.** This RFC covers building,
  generating, and packaging symbols into artifact archives only. Hosting a Windows
  Symbol Server, hosting a Linux `debuginfod` service, generating the database/index
  structures those systems require, and the infrastructure to push symbols to them are
  all out of scope and are considered as future enhancements.
- **Non-CMake sysdeps.** System dependencies built through autotools or meson (e.g.
  `elfutils`, `libdrm`, `numactl`) are not covered by the CMake-based debug pipeline and
  are out of scope; their debug info is left as the upstream build produces it.
- **macOS**, which is treated as best-effort and not gated by this RFC.

## Background: Current State

TheRock already contains partial, Linux-only infrastructure. Understanding it is
essential because this RFC formalizes and extends it rather than replacing it.

| Mechanism                    | Location                                                                                         | Behavior today                                                                                                                                                                        |
| :--------------------------- | :----------------------------------------------------------------------------------------------- | :------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| `THEROCK_SPLIT_DEBUG_INFO`   | `CMakeLists.txt`                                                                                 | Linux-only. Splits debug info into `.build-id` files and strips primary binaries via `cmake --install --strip`.                                                                       |
| `THEROCK_MINIMAL_DEBUG_INFO` | `CMakeLists.txt`, `cmake/therock_subproject.cmake`                                               | Appends `-g1 -gz` to `*_FLAGS_DEBUG` and `*_FLAGS_RELWITHDEBINFO` **only** — never to `Release`.                                                                                      |
| Build-id split pipeline      | `cmake/therock_global_post_subproject.cmake`, `cmake/therock_install_linux_build_id_files.cmake` | Forces `-Wl,--build-id`, runs `objcopy --only-keep-debug` to emit `.build-id/XX/YYY.debug`.                                                                                           |
| `dbg` artifact component     | `build_tools/_therock_utils/artifact_builder.py`                                                 | Default include pattern `.build-id/**/*.debug` (Linux only).                                                                                                                          |
| MSVC embedded debug          | `cmake/therock_compiler_config.cmake`                                                            | Sets `CMAKE_MSVC_DEBUG_INFORMATION_FORMAT=Embedded` (i.e. `/Z7`, not `/Zi`) for `Debug,RelWithDebInfo` — required for ccache/sccache.                                                 |
| Release presets              | `CMakePresets.json`, CI workflows                                                                | `linux-release-package` uses `CMAKE_BUILD_TYPE=RelWithDebInfo` (`-O2`) to obtain symbols but is not consistently consumed by CI; `windows-release` uses `Release` with no debug info. |

### Key gaps

1. **Release does not carry symbols.** The only way the current Linux release obtains
   debug info is by selecting `CMAKE_BUILD_TYPE=RelWithDebInfo`, which lowers
   optimization to `-O2` and changes codegen relative to the shipped `-O3 -DNDEBUG`
   Release. This violates the requirement that release builds keep full optimization.
1. **No Windows debug pipeline.** PDBs are not generated, collected, or packaged in
   release; the `dbg` component default pattern is Linux-only.
1. **No detail-level control.** Only a single `-g1` "minimal" path exists; there is no
   full (`-g2`) level and no public/private PDB distinction.
1. **`RelWithDebInfo` uses its own path** (CMake's built-in `-g`) rather than a shared
   `THEROCK_` mechanism, so behavior diverges across OSes and build types.
1. **No per-component size control.** A global-only switch would force debug info into
   every component, even when one component (notably the bootstrapped compiler) would
   dominate symbol storage. Release policy needs a per-component opt-out without
   changing that component's build type or packaged file set.
1. **`dbg` is sliced but not shipped.** Debug artifacts are not yet emitted as
   dedicated debug tarballs or deb `-dbgsym` / rpm `-debuginfo` packages, and several
   relevant artifact descriptors do not declare a `dbg` component at all.

## Design

### 1. Unified, OS-agnostic CMake mechanism

Define three project-wide settings that apply uniformly to every build type, including
`Release`:

- `THEROCK_GENERATE_DEBUG_INFO` (`STRING`, one of empty, `ON`, or `OFF`; default
  empty): tri-state master control. Empty means **auto**: enable generation for
  `Debug` and `RelWithDebInfo`, and disable it for other build types. Explicit `ON` or
  `OFF` becomes the default for every subproject.
- `THEROCK_DEBUG_INFO_LEVEL` (`STRING`, one of `minimal`, `full`, or `extra`; default
  `full`): selects the amount of debug information. Windows supports `minimal` and
  `full`; `extra` emits a warning and is treated as `full`.
- `THEROCK_SPLIT_DEBUG_INFO` (`BOOL`): extracts debug info into separate files and
  strips primary binaries. It defaults to `OFF` on Linux. Windows always uses split
  symbols because PDBs cannot be embedded in the image, so it defaults to `ON`; an
  explicit `OFF` emits a warning and is overridden to `ON`.

Each setting has a per-subproject counterpart, keyed by the exact TheRock subproject
target name:

- `{project}_GENERATE_DEBUG_INFO`
- `{project}_DEBUG_INFO_LEVEL`
- `{project}_SPLIT_DEBUG_INFO`

This follows the existing `{project}_BUILD_TYPE` convention. For each subproject, the
effective settings are resolved as follows:

1. Generation uses `{project}_GENERATE_DEBUG_INFO` when explicitly defined.
1. Otherwise, an explicit global `THEROCK_GENERATE_DEBUG_INFO=ON|OFF` is inherited.
1. Otherwise, generation follows the effective build type: `{project}_BUILD_TYPE` when
   set, else `CMAKE_BUILD_TYPE`; `Debug`/`RelWithDebInfo` resolve to `ON`, all other
   build types to `OFF`.
1. Level uses `{project}_DEBUG_INFO_LEVEL` when defined, else
   `THEROCK_DEBUG_INFO_LEVEL` (`full` by default).
1. On non-Windows platforms, splitting uses `{project}_SPLIT_DEBUG_INFO` when defined,
   else `THEROCK_SPLIT_DEBUG_INFO`. On Windows it always resolves to `ON`.

Boolean spellings are validated and normalized to `ON`/`OFF`. An invalid level is a
configuration error. A per-subproject level or split setting whose generation resolves
to `OFF` emits a warning because that value is ignored.

On Linux, a generated `CMAKE_USER_MAKE_RULES_OVERRIDE` adjusts the composed
`CMAKE_<LANG>_FLAGS_<CONFIG>_INIT` values after platform defaults are known and before
the cache entries are created. It strips CMake's built-in `-g` selection and appends the
resolved level to every standard configuration, including `Release`, while preserving
that configuration's optimization flags.

| OS / compiler     | `minimal`                                      | `full`                         | `extra`                          |
| :---------------- | :--------------------------------------------- | :----------------------------- | :------------------------------- |
| Linux (GCC/Clang) | `-g1 -gz`                                      | `-g2 -gz`                      | `-g3 -gz`                        |
| Windows (MSVC)    | public PDB (`/Z7` + `/DEBUG` + `/PDBSTRIPPED`) | private PDB (`/Z7` + `/DEBUG`) | unsupported; warning, use `full` |

`-gz` keeps the separated `.debug` files compressed. `/Z7` (object-embedded debug,
not `/Zi`) is mandatory for compiler-cache compatibility (see §4).

### 2. `RelWithDebInfo` unification

`RelWithDebInfo` retains its own optimization level; only its debug-info handling
changes. With `THEROCK_GENERATE_DEBUG_INFO` left in its empty auto state, a
`RelWithDebInfo` project resolves generation to `ON`. The build clears CMake's built-in
`-g` from the composed flags and injects the resolved level through the unified
mechanism. Linux retains the default `THEROCK_SPLIT_DEBUG_INFO=OFF` for local debugging;
Windows still emits a separate PDB because splitting is mandatory there. An explicit
global or per-project generation setting takes precedence over this build-type default.

### 3. Linux pipeline

The existing pipeline is retained and made Release-clean:

1. Targets are compiled with `-g{level} -gz` injected into the active config and linked
   with `-Wl,--build-id` (already forced in `therock_global_post_subproject.cmake`).
1. At install, `objcopy --only-keep-debug` extracts symbols from the unstripped
   build-tree binary into `.build-id/XX/YYY.debug`
   (`therock_install_linux_build_id_files.cmake`); the staged binary is stripped via
   `cmake --install --strip`.
1. The `dbg` artifact component collects `.build-id/**/*.debug`.

Extraction and stripping run when a subproject's effective
`GENERATE_DEBUG_INFO=ON` **and** effective `SPLIT_DEBUG_INFO=ON`, independent of build
type. Generation with splitting disabled retains embedded debug information on Linux,
which remains the default local-development behavior.

**Non-CMake dependencies.** The build-id split hook operates on CMake targets, so
sysdeps built through autotools or meson (e.g. `elfutils`, `libdrm`, `numactl`) are not
seen by it. Their debug output remains best-effort and is omitted from the `dbg`
deliverable when the upstream build does not provide compatible symbols. This is why
the guaranteed coverage is scoped to CMake-built host binaries.

### 4. Windows pipeline

Windows gains a first-class PDB pipeline:

1. Each subproject whose effective generation setting is `ON` receives
   `CMAKE_MSVC_DEBUG_INFORMATION_FORMAT=Embedded` (`/Z7`) through its generated
   toolchain and `-DCMAKE_POLICY_DEFAULT_CMP0141=NEW` on its configure command. A
   subproject that resolves generation to `OFF` receives an empty format and no policy
   override, preventing a global `Embedded` baseline from leaking into an opted-out
   project. `/Z7` rather than `/Zi` is required so `ccache`/`sccache` can cache objects
   (see [ROCm/TheRock#4419](https://github.com/ROCm/TheRock/pull/4419)).
1. The linker is invoked with `/DEBUG:FULL`, which emits the full private PDB at
   `<dir>/<name>.pdb` next to each DLL/EXE. For `minimal`, the linker is additionally
   passed `/PDBSTRIPPED:<dir>/stripped/<name>.pdb` to emit a stripped public PDB into a
   sibling sub-folder (the full PDB is always produced and cannot be suppressed; emitting
   the stripped PDB into its own folder lets it keep the same `<name>.pdb` basename
   without a filename suffix).
1. The level is enforced when staging the `dbg` artifact, not at generation: the `dbg`
   component places only the PDB selected by `THEROCK_DEBUG_INFO_LEVEL` — the full PDB
   for `full`, or the `stripped/` PDB for `minimal` — staged under its canonical
   `<name>.pdb` name, with the unselected PDB excluded. PDBs are inherently separate
   files, so this is the Windows analogue of the Linux `.build-id/**/*.debug` slice.

### 5. Per-component controls and release size policy

Debug-info volume varies greatly by component. The bootstrapped compiler can dominate
symbol storage, so enabling full debug information globally without a project-level
escape hatch is not practical for every release configuration. Package and development
artifact contents are unchanged by this RFC; size is controlled at debug-info generation
time instead.

Release presets enable full debug information globally and may disable generation for
selected high-volume components. The initial release-packaging policy is:

```cmake
-DTHEROCK_GENERATE_DEBUG_INFO=ON
-DTHEROCK_DEBUG_INFO_LEVEL=full
-DTHEROCK_SPLIT_DEBUG_INFO=ON
-Damd-llvm_GENERATE_DEBUG_INFO=OFF
```

Under this preset, all other subprojects inherit full debug information, while
`amd-llvm` is compiled and linked without host debug symbols and contributes no files to
the debug deliverable. `amd-llvm_GENERATE_DEBUG_INFO=OFF` is not a hardcoded component
default: users and other presets can enable it, and other components can use the same
opt-out or select a lower `{project}_DEBUG_INFO_LEVEL` as storage and debugging
requirements evolve. This size-control requirement is the primary reason per-component
settings are part of the core design.

### 6. Packaging

Debug symbols are surfaced as **separate deliverables** in the tarball and native
packaging channels, leaving runtime and development packages stripped:

- **Tarballs:** the `dbg` artifact component is archived and published as a distinct
  debug tarball per build, parallel to the runtime/devel tarballs.
- **Native deb/rpm:** debug content is emitted as the platform-standard companion
  package — `-dbgsym` (Debian) and `-debuginfo` (RPM) — sourced from the `dbg`
  component. `package.json` entries gain a `dbg` component mapping, and the
  `dh_strip`/`debugedit` interactions noted in
  `build_tools/packaging/linux/deb_package.py`
  ([issue #4047](https://github.com/ROCm/TheRock/issues/4047)) must be re-validated so the
  split files are not corrupted.

Every relevant in-scope artifact descriptor that produces CMake-built shippable
binaries must declare a `[components.dbg."<stage>"]` entry so its debug output is
captured rather than silently dropped.

### 7. CI integration

These changes become the default for nightly and release builds:

- **Linux release preset:** update `CMakePresets.json` to use
  `CMAKE_BUILD_TYPE=Release` and set
  `THEROCK_GENERATE_DEBUG_INFO=ON`, `THEROCK_DEBUG_INFO_LEVEL=full`, and
  `THEROCK_SPLIT_DEBUG_INFO=ON`. Existing per-subproject `*_BUILD_TYPE=Release`
  overrides remain. Set `amd-llvm_GENERATE_DEBUG_INFO=OFF` because the bootstrapped
  compiler's debug information would otherwise dominate release symbol size. This
  override belongs to the release preset and remains user-overridable.
- **Linux CI wiring:** update active nightly/release workflows to consume the Linux
  release preset or pass the same effective values. Updating the preset alone is not
  sufficient while preset use is inconsistent; this adoption gap is tracked by
  [#1781](https://github.com/ROCm/TheRock/issues/1781) and
  [#5453](https://github.com/ROCm/TheRock/issues/5453).
- **Windows release preset:** keep `CMAKE_BUILD_TYPE=Release`, set generation and level
  to `ON`/`full`, and rely on the Windows rule that splitting is always `ON`. Set
  `amd-llvm_GENERATE_DEBUG_INFO=OFF` in the preset for the same release symbol-size
  constraint; users can override it.
- **ccache/sccache:** verify hit rates are unaffected. Linux `-g`/`-gz` hashing is
  cache-safe; Windows correctness depends on `/Z7` (above). No change to
  `setup_ccache.py` / `setup_sccache_rocm.py` is expected beyond validation.

## Implementation Plan

### Phase 1: Unified CMake mechanism

- Add tri-state `THEROCK_GENERATE_DEBUG_INFO`, three-level
  `THEROCK_DEBUG_INFO_LEVEL`, and platform-defaulted `THEROCK_SPLIT_DEBUG_INFO`.
- Add the three `{project}_*` overrides and central resolution/validation with the
  precedence defined in §1.
- Resolve settings before generating each subproject's init, toolchain, and make-rules
  override files; route every build type, including `Release` and `RelWithDebInfo`,
  through those effective settings.

### Phase 2: Per-OS pipelines

- Make the Linux build-id split run whenever effective generation and splitting are
  both `ON`, including under `Release`.
- Implement the Windows PDB pipeline and add the `**/*.pdb` `dbg` default pattern.
- Validate ccache/sccache behavior on both OSes.

### Phase 3: Per-component size controls

- Validate generation, level, and split overrides independently for representative
  subprojects on Linux and Windows.
- Set `amd-llvm_GENERATE_DEBUG_INFO=OFF` in release presets while preserving user
  override behavior.
- Verify that changing an override rewrites the generated configuration inputs,
  changes the component fingerprint, and automatically reconfigures that subproject on
  the next build after top-level CMake configuration.

### Phase 4: Packaging

- Declare `dbg` components on all relevant artifact descriptors.
- Emit separate debug tarballs and deb `-dbgsym`/rpm `-debuginfo` packages.

### Phase 5: Preset and CI default enablement

- Update Linux and Windows release presets.
- Wire the Linux release settings into active nightly/release workflows by consuming
  the preset or passing equivalent values, and verify the effective CI configuration.
- Publish debug artifacts to the debug bucket.
- Enable by default for nightly/release.

## Risks and Considerations

- **Package size / storage.** Full debug symbols are large. Mitigated by always
  splitting for release packaging, compressing (`-gz`), shipping debug separately, and
  disabling generation for selected high-volume components such as `amd-llvm`.
- **Configuration precedence.** Global, per-project, and build-type-derived defaults
  can be confusing unless resolution is deterministic. The precedence in §1 is
  centralized, invalid values fail configuration, and ignored or unsupported settings
  emit warnings.
- **Stripping correctness** for unconventional ELF layouts (deb `debugedit`,
  issue 4047) and for kpack-split artifacts (RFC0008) must be re-verified.
- **Build time / cache.** `/Zi` would break Windows caching; `/Z7` is mandatory.
