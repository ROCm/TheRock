# Tarball Packaging

Tarballs are the foundational packaging format in TheRock. They provide a flat,
self-contained ROCm SDK directory that other packaging formats (Python wheels,
DEB, RPM) are derived from.

## Overview

A ROCm SDK tarball is a compressed archive containing the complete, flattened
install tree from a build. The naming convention is:

```
therock-dist-{platform}-{gfx_family}-{version}.tar.gz
```

For example:

```
therock-dist-linux-gfx94X-dcgpu-7.12.0a20260310.tar.gz
```

Where:
- `platform`: `linux` or `windows`
- `gfx_family`: GPU architecture family (e.g. `gfx94X-dcgpu`, `gfx110X-all`)
- `version`: ROCm version with channel suffix (e.g. `7.12.0a20260310` for
  nightly)

## Pre-submit vs Nightly Tarballs

TheRock produces two distinct types of archives depending on the CI context.

### Pre-submit: Per-component Artifact Archives

When a PR is submitted, CI builds individual components and produces
**per-component artifact archives**. These are fine-grained `.tar.xz` files,
one per artifact-component-target combination:

```
build/artifacts/
в”њв”Ђв”Ђ base_lib_generic.tar.xz
в”њв”Ђв”Ђ base_dev_generic.tar.xz
в”њв”Ђв”Ђ base_run_generic.tar.xz
в”њв”Ђв”Ђ core-hip_lib_gfx942.tar.xz
в”њв”Ђв”Ђ core-hip_dev_gfx942.tar.xz
в”њв”Ђв”Ђ blas_lib_gfx942.tar.xz
в”њв”Ђв”Ђ blas_dev_gfx942.tar.xz
в””в”Ђв”Ђ ...
```

These archives:
- Preserve the full build-tree layout (paths like
  `core/clr/build/stage/lib/libamdhip64.so`).
- Can bootstrap multi-stage builds by extracting into a build directory.
- Are uploaded as GitHub Actions artifacts for downstream jobs.
- Are **not** intended for end-user installation.

The archives are generated by the `therock-archives` CMake target, which
invokes `fileset_tool.py artifact-archive` for each component.

### Nightly: Unified SDK Tarball

Nightly (and release) builds produce a **single unified tarball** containing
the complete ROCm SDK. This is the `build/dist/rocm/` directory compressed
into a single `.tar.gz`:

```
therock-dist-linux-gfx94X-dcgpu-7.12.0a20260310.tar.gz
в””в”Ђв”Ђ bin/
    include/
    lib/
    libexec/
    share/
    .info/
```

This tarball:
- Contains a flat install tree (all component stage directories merged).
- Follows the traditional `/opt/rocm/` layout.
- Is ready for direct extraction and use.
- Is uploaded to S3 (e.g. `https://rocm.nightlies.amd.com/tarball/`).

## Tarball Generation Process

### Step 1: Build and Stage

Each sub-project in the CMake build produces an install tree under its
`stage/` directory:

```
build/
в”њв”Ђв”Ђ base/rocm-core/build/stage/
в”‚   в”њв”Ђв”Ђ lib/librocm-core.so
в”‚   в”њв”Ђв”Ђ .info/version.json
в”‚   в””в”Ђв”Ђ libexec/rocm-core-healthcheck
в”њв”Ђв”Ђ core/clr/build/stage/
в”‚   в”њв”Ђв”Ђ lib/libamdhip64.so
в”‚   в”њв”Ђв”Ђ include/hip/hip_runtime.h
в”‚   в””в”Ђв”Ђ bin/hipcc
в””в”Ђв”Ђ ...
```

### Step 2: Artifact Population

The `therock_provide_artifact()` CMake function, driven by TOML descriptors,
slices each sub-project's `stage/` into artifact component directories:

```cmake
therock_provide_artifact(base
  TARGET_NEUTRAL
  DESCRIPTOR artifact.toml
  COMPONENTS dev doc lib run test
  SUBPROJECT_DEPS therock-half therock-rocm-core therock-rocprofiler-register ...
)
```

The TOML descriptor (`artifact.toml`) defines which files from each stage
directory go into which component using ant-style glob patterns:

```toml
# rocm-core
[components.lib."base/rocm-core/stage"]
include = [".info/**"]

[components.run."base/rocm-core/stage"]
include = "libexec/**"

[components.dev."base/rocm-core/stage"]
include = "lib/rocmmod"
```

This produces directories under `build/artifacts/`:

```
build/artifacts/
в”њв”Ђв”Ђ base_lib_generic/
в”‚   в”њв”Ђв”Ђ artifact_manifest.txt
в”‚   в””в”Ђв”Ђ base/rocm-core/build/stage/lib/librocm-core.so
в”њв”Ђв”Ђ base_dev_generic/
в”‚   в””в”Ђв”Ђ base/rocm-core/build/stage/include/...
в””в”Ђв”Ђ ...
```

### Step 3: Flattening to Distribution

All artifact directories are flattened into `build/dist/rocm/` by the
`dist-rocm` target, using `fileset_tool.py artifact-flatten`. This reads
each `artifact_manifest.txt` and merges the files, collapsing the deep
build-tree structure into a flat install tree:

```
build/dist/rocm/
в”њв”Ђв”Ђ bin/hipcc
в”њв”Ђв”Ђ include/hip/hip_runtime.h
в”њв”Ђв”Ђ lib/libamdhip64.so
в”њв”Ђв”Ђ share/therock/therock_manifest.json
в””в”Ђв”Ђ ...
```

### Step 4: Archive Creation

For **per-component archives** (pre-submit): `fileset_tool.py artifact-archive`
compresses each artifact directory into `.tar.xz` with SHA256 checksums.
Compression level defaults to 2 for speed in CI.

For the **unified SDK tarball** (nightly/release): The CI workflow runs
`tar cfz` on the `dist/rocm/` directory, producing the final
`therock-dist-linux-*.tar.gz`.

## What Goes Into a Tarball

### Inclusion Rules

The artifact TOML descriptors define what files are included. Each component
type has default patterns, and descriptors can add custom `include` patterns:

1. **Default patterns** вЂ” Each component type (`lib`, `dev`, `run`, etc.) has
   built-in include/exclude patterns in `fileset_tool.py`.
2. **Descriptor includes** вЂ” TOML entries can add `include` patterns for files
   that don't match defaults (e.g. `include = [".info/**"]` for metadata).
3. **Force includes** вЂ” `force_include` patterns override exclusion rules.
4. **Optional entries** вЂ” `optional = true` allows a stage directory to be
   absent without error (for platform-conditional sub-projects).

### Exclusion Rules

1. **Default exclusions** вЂ” Each component type excludes files that belong to
   other components (e.g. `lib` excludes headers, `dev` excludes `.so` files).
2. **Descriptor excludes** вЂ” TOML entries can add `exclude` patterns.
3. **Unmatched files** вЂ” Files not matching any component's patterns generate
   warnings during `fileset_tool.py artifact` to catch regressions.

### Component Type Patterns

| Component | Default Includes | Default Excludes |
| --- | --- | --- |
| `lib` | `lib/**/*.so*`, `lib/**/*.dylib`, runtime data | Headers, static libs, cmake configs |
| `dev` | `include/**`, `lib/**/*.a`, `lib/**/cmake/**`, `lib/**/pkgconfig/**` | Shared libs (except import libs) |
| `run` | `bin/**`, `libexec/**` | Libraries, headers |
| `dbg` | `lib/**/*.debug`, debug symbol files | Everything else |
| `doc` | `share/**/*license*`, `share/**/*LICENSE*`, docs | Binaries, libraries |
| `test` | `share/**/tests/**`, test binaries | Production files |

### Component Placement Map

This table shows which ROCm components contribute files to each top-level
directory, and the corresponding artifact that provides them:

| Directory | Artifact Source | Contents | GPU-Specific? |
| --- | --- | --- | --- |
| `bin/` | `base` (run), `core-hip` (run), `amd-llvm` (run), `rocprofiler-sdk` (run) | CLI tools, compilers, profilers | No |
| `include/hip/` | `core-hip` (dev) | HIP runtime headers | No |
| `include/rocblas/` | `blas` (dev) | rocBLAS headers | No |
| `include/miopen/` | `MIOpen` (dev) | MIOpen headers | No |
| `lib/libamdhip64.so` | `core-hip` (lib) | HIP runtime | No (host code) |
| `lib/librocblas.so` | `blas` (lib) | Linear algebra | Yes (device code) |
| `lib/librocfft.so` | `fft` (lib) | FFT library | Yes (device code) |
| `lib/libMIOpen.so` | `MIOpen` (lib) | Deep learning primitives | Yes (device code) |
| `lib/librccl.so` | `rccl` (lib) | Collective communications | Yes (device code) |
| `lib/librocrand.so` | `rand` (lib) | Random number generation | Yes (device code) |
| `lib/cmake/` | Various (dev) | CMake find-package configs | No |
| `lib/rocm_sysdeps/` | `sysdeps` (lib) | Private system deps (zlib, elfutils, etc.) | No |
| `libexec/` | `base` (run) | Internal executables | No |
| `llvm/` | `amd-llvm` (lib, dev, run) | Full LLVM/Clang toolchain | No |
| `share/therock/` | `base` (lib) | Build manifest, metadata | No |
| `share/doc/` | Various (doc) | License and doc files | No |
| `.info/` | `base` (lib) | Version info | No |

**Key design points:**

- **Target-neutral artifacts** (`_generic` suffix) contain host code and
  headers that work across GPU families.
- **Target-specific artifacts** (e.g. `_gfx942`) contain GPU device code
  (kernel objects, ISA-specific libraries).
- **System dependencies** (`sysdeps`) use private SONAMEs
  (`librocm_sysdeps_*.so`) to avoid conflicts with system libraries.
- **LLVM** gets its own `llvm/` subtree rather than mixing into `lib/` to
  keep the compiler toolchain self-contained.
- The `therock_manifest.json` in `share/therock/` records exact commit SHAs
  for all submodules, enabling build reproducibility.

## Installation

### Manual Extraction

```bash
mkdir therock-tarball && cd therock-tarball
wget https://rocm.nightlies.amd.com/tarball/therock-dist-linux-gfx94X-dcgpu-7.12.0a20260310.tar.gz
mkdir install && tar -xf *.tar.gz -C install
```

### Automated Installation

The
[`install_rocm_from_artifacts.py`](/build_tools/install_rocm_from_artifacts.py)
script supports several installation modes:

```bash
# Install latest nightly for a GPU family
python build_tools/install_rocm_from_artifacts.py \
    --latest-release \
    --artifact-group gfx94X-dcgpu \
    --output-dir ./therock-build

# Install a specific release version
python build_tools/install_rocm_from_artifacts.py \
    --release 7.12.0a20260310 \
    --artifact-group gfx94X-dcgpu \
    --output-dir ./therock-build

# Install from a specific CI run (per-component artifacts)
python build_tools/install_rocm_from_artifacts.py \
    --run-id 21440027240 \
    --artifact-group gfx94X-dcgpu \
    --output-dir ./therock-build
```

### Post-Installation

After extraction, configure your environment:

```bash
export ROCM_HOME=$(pwd)/install
export PATH="$ROCM_HOME/bin:$PATH"
export LD_LIBRARY_PATH="$ROCM_HOME/lib:$LD_LIBRARY_PATH"

# Verify the installation
./install/bin/rocminfo
```

Build provenance is available in the manifest:

```bash
cat install/share/therock/therock_manifest.json
```
